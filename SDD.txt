3. Database Schema

General Supabase Rules
- Auth identity source: `auth.users`
- All app-owned tables use primary keys
- User-owned rows use `user_id` or `id` linked to `auth.users(id)`
- Foreign keys use `on delete cascade` unless business logic requires otherwise
- `created_at` defaults to `now()` where applicable
- `updated_at` defaults to `now()` where applicable on mutable tables
- Row ownership is enforced by the authenticated user
- RLS enabled on all public app tables

Profiles
- Primary key: `id uuid primary key references auth.users(id) on delete cascade`
- Columns:
  - `email text`
  - `username text unique`
  - `created_at timestamptz not null default now()`
  - `updated_at timestamptz not null default now()`
- Constraints:
  - optional username with length/format validation
- Ownership:
  - row owned by authenticated user where `profiles.id = auth.uid()`
- RLS:
  - users can select, insert, update, and delete only their own profile row

Preferences
- Primary key: `id uuid primary key default gen_random_uuid()`
- Foreign keys:
  - `user_id uuid not null unique references auth.users(id) on delete cascade`
- Columns:
  - `favorite_colors text[] not null default '{}'`
  - `disliked_colors text[] not null default '{}'`
  - `preferred_styles text[] not null default '{}'`
  - `sizes jsonb not null default '{}'::jsonb`
  - `created_at timestamptz not null default now()`
  - `updated_at timestamptz not null default now()`
- Constraints:
  - one preferences row per user via unique `user_id`
- Ownership:
  - row owned by authenticated user where `preferences.user_id = auth.uid()`
- RLS:
  - users can select, insert, update, and delete only their own preferences row

ClothingItems
- Primary key: `id uuid primary key default gen_random_uuid()`
- Foreign keys:
  - `user_id uuid not null references auth.users(id) on delete cascade`
- Columns:
  - `name text`
  - `image_path text not null`
  - `category text`
  - `subcategory text`
  - `color text`
  - `secondary_colors text[]`
  - `pattern text`
  - `material text`
  - `formality text`
  - `fit text`
  - `season_tags text[]`
  - `occasion_tags text[]`
  - `style_tags text[]`
  - `weather_tags text[]`
  - `brand text`
  - `ai_raw_output jsonb`
  - `created_at timestamptz not null default now()`
  - `updated_at timestamptz not null default now()`
  - `last_worn date`
  - `is_favorite boolean not null default false`
  - `notes text`
- Ownership:
  - row owned by authenticated user where `clothing_items.user_id = auth.uid()`
- RLS:
  - users can select, insert, update, and delete only their own clothing items
- Storage:
  - clothing images stored in Supabase Storage
  - recommended private bucket: `clothing`
  - recommended object path pattern: `<user_id>/<item_id>/<filename>`
  - `image_path` stores storage object path, not a public URL

Outfits
- Primary key: `id uuid primary key default gen_random_uuid()`
- Foreign keys:
  - `user_id uuid not null references auth.users(id) on delete cascade`
- Columns:
  - `name text`
  - `occasion text`
  - `notes text`
  - `is_favorite boolean not null default false`
- Ownership:
  - row owned by authenticated user where `outfits.user_id = auth.uid()`
- RLS:
  - users can select, insert, update, and delete only their own outfits

OutfitItems
- Primary key: `id uuid primary key default gen_random_uuid()`
- Foreign keys:
  - `outfit_id uuid not null references outfits(id) on delete cascade`
  - `clothing_item_id uuid not null references clothing_items(id) on delete cascade`
- Columns:
  - `created_at timestamptz not null default now()`
- Constraints:
  - unique `(outfit_id, clothing_item_id)` to prevent duplicate item membership in the same outfit
- Ownership:
  - row access derived from ownership of the referenced outfit
- RLS:
  - users can select, insert, update, and delete only outfit_items connected to their own outfits

WearHistory
- Primary key: `id uuid primary key default gen_random_uuid()`
- Foreign keys:
  - `user_id uuid not null references auth.users(id) on delete cascade`
  - `outfit_id uuid null references outfits(id) on delete set null`
- Columns:
  - `worn_date date not null`
  - `notes text`
  - `created_at timestamptz not null default now()`
- Constraints:
  - optional unique logging rule can prevent duplicate outfit entries for the same user on the same date
- Ownership:
  - row owned by authenticated user where `wear_history.user_id = auth.uid()`
- RLS:
  - users can select, insert, update, and delete only their own wear history rows

Feedback
- Primary key: `id uuid primary key default gen_random_uuid()`
- Foreign keys:
  - `user_id uuid not null references auth.users(id) on delete cascade`
  - `outfit_id uuid not null references outfits(id) on delete cascade`
- Columns:
  - `feedback_type text not null`
  - `feedback_note text`
  - `created_at timestamptz not null default now()`
- Constraints:
  - recommended check constraint on `feedback_type` such as `liked`, `disliked`, `too_formal`, `too_casual`, `too_hot`, `too_cold`
- Ownership:
  - row owned by authenticated user where `feedback.user_id = auth.uid()`
- RLS:
  - users can select, insert, update, and delete only their own feedback rows

Storage Buckets
- `clothing` bucket
  - stores uploaded clothing photos
  - intended to be private
  - users should only access objects in their own folder path
  - access controlled via RLS policies on `storage.objects`
  - users can only read/write files within their own folder (`<user_id>/...`)
  - recommended allowed file types: jpg, jpeg, png, webp, heic
  - images accessed via signed URLs (private bucket)
  - `image_path` is used to generate signed URLs at runtime
- optional `avatars` bucket
  - stores profile images if profile avatars are added later
  - should also use user-scoped paths and RLS-backed storage policies
  - should follow same user-scoped folder pattern (`<user_id>/...`)
  - access controlled via RLS policies

5. Data Flow

Upload Flow:
User -> Upload Image -> API -> Storage -> Save Metadata -> Confirm Tags -> Save Item

Recommendation Flow (Stylist tab, client-side):
1. Tab focus  ->  Supabase: SELECT * FROM clothing_items WHERE user_id = ?
                            SELECT preferred_styles FROM preferences WHERE user_id = ?
2. First load ->  Pre-select up to 2 vibe chips from preferred_styles
3. User selects occasion/vibe chips (or submits free-text prompt)
4. Client-side scoring pipeline (no network call):
     a. Seeded-shuffle the item pool (deterministic per seed value)
     b. Score each item:
          +3 per occasion keyword hit (item.occasion_tags vs OCCASION_TAGS map)
          +2 per vibe keyword hit    (item.style_tags vs VIBE_TAGS map)
          +1 if item.is_favorite
     c. Sort descending by score; group into structural buckets:
          dress | top | bottom | shoes | outer | accessory | other
     d. Greedy combo selection â€” try templates in order of preference:
          [top, bottom, shoes] -> [dress, shoes] -> [top, bottom, outer]
          -> [top, bottom] -> [dress]
          Pick the highest-scoring valid template; mark chosen items as used;
          repeat up to 3 times to produce up to 3 outfit cards.
     e. Normalise raw score to confidence %:
          confidence = clamp(62 + (rawScore / maxPossible) * 35, 62, 97)
     f. Fallback: if no template could be satisfied, show top 4 scored items
5. SuggestionCard renders -> per-item: Supabase Storage createSignedUrl(image_path)
6. User taps "Regenerate" -> increment seed -> step 4 reruns with a new shuffle
7. "Wear today" (not yet wired) -> will INSERT into wear_history, UPDATE last_worn

Feedback Flow (not yet wired to backend):
User thumbs-up / thumbs-down / bookmark -> local state only (future: INSERT into feedback table)
